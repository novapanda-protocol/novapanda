# Troodon · 架构设计

> **配套**：《设计-协议v0与模拟舱》（v0 规范与范围）、《详细设计》（数据模型/API/时序）。
> **本篇职责**：系统架构、组件边界、联邦部署、数据流、**信任与安全架构**、中立性如何由架构保证、演进路线。
> **约束来源**：设计底线（普适通用 · 降低接入难度 · 要标准 · 要开放）与信仰（万物交换、先交付后付款、不铸币、不被统治）。

---

## 1. 架构目标与约束（从信仰推导）

| 信仰/底线 | 架构强约束 |
|-----------|-----------|
| 普适通用 | 核心与业务/框架/语言/链/币**解耦**；领域扩展只动「资源本体」，不动核心协议 |
| 降低接入难度 | 多接入面（Skill/MCP/A2A/SDK/适配器/裸 HTTP）收敛到**同一套 VDC+状态机** |
| 要标准 | 核心**最小且稳定**、可版本化、可认证；多实现可互通（像 HTTP） |
| 要开放 | **不信任任何单一节点**：一切声明可被任何人独立复验；多节点联邦；无单一闸门 |
| 不铸币/不收租 | 结算是**外置可换的腿**，核心不持有价值、不强制结算介质 |
| 可退出 | 数据可导出、可自托管；离开任一节点不丢凭证与信誉 |

> **第一性原则：不信任节点（trust-minimized）。** 即便是我们自己运营的零号节点，也**不能伪造**任何交割——因为每张 VDC、每条信誉都由当事方签名且可被第三方独立复算。这是"开放、不被统治"在架构层的兑现。

## 2. 分层架构

```
┌──────────────────────────────────────────────────────────────┐
│ L5 服务层  Exchange Node（可多实例·人人可自建·可被替代）          │  ← 身体/可运营
│            撮合 · 状态机编排 · 验收调度 · 结算调度 · 信誉账本 · 注册表 │
├──────────────────────────────────────────────────────────────┤
│ L4 接入层  Skill / MCP / A2A / SDK / 适配器 / 裸 HTTP             │  ← 仅做"翻译"
├──────────────────────────────────────────────────────────────┤
│ L3 协议层  交换状态机 + 协议消息（PROPOSED…SETTLED/REJECTED）      │  ← 共域·稳定
├──────────────────────────────────────────────────────────────┤
│ L2 凭证层  VDC 结构 + 规范化(JCS) + 签名(Ed25519) + 哈希(SHA-256)  │  ← 共域·稳定
├──────────────────────────────────────────────────────────────┤
│ L1 语义层  资源本体(Ontology) + 规则注册表(Rule Registry)         │  ← 共域·可扩展
├──────────────────────────────────────────────────────────────┤
│ L0 身份层  Ed25519 keypair / agent_id / did:troodon  │
└──────────────────────────────────────────────────────────────┘
   旁路（外置、可插拔、可竞争）：
   ┌ 结算适配器 Settlement：Mock / x402 / AP2 / 法币持牌伙伴
   ├ 验收器 Verifier：Schema(确定性) / LLM 裁判 / 未来 ZK·TEE
   └ 见证 Witness（未来）：第三方背书 / 质押罚没
```

**关键边界**：
- **L0–L3 是"灵魂"**：开放许可、最小、稳定、可版本化——**任何人可独立实现**。
- **L4 只做翻译**：把外部世界映射到 L3 生命周期，**绝不把框架/传输写进核心**。
- **L5 是"身体"**：可运营、可盈利、**但可被替代**；多节点联邦，无单点。
- **旁路可竞争**：结算/验收/见证都是可插拔市场，不由核心垄断。

> 这张分层图就是"灵魂 vs 身体"在技术上的落地：L0–L3 归共域，L5+旁路是可运营、可竞争的市场。

## 3. 组件视图

```
        Client Agent A                         Provider Agent B
        (任意语言/框架)                          (任意语言/框架)
            │  接入面(L4)                              │  接入面(L4)
            ▼                                          ▼
   ┌───────────────────────────── Exchange Node (L5) ─────────────────────────────┐
   │                                                                               │
   │  [API 网关] ──► [状态机引擎] ──► [Exchange Store(持久化)]                        │
   │       │              │                                                        │
   │       │              ├──► [验收调度] ──► Verifier(旁路)  ←── 可被任何人复跑      │
   │       │              ├──► [结算调度] ──► Settlement(旁路) ←── 不绑币·可换        │
   │       │              └──► [信誉账本] (只追加·哈希链·签名) ←── 可跨节点复验        │
   │       │                                                                       │
   │  [注册表服务] 资源本体 + 规则（只读·开放）                                        │
   │  [身份/验签] Ed25519 验签 · nonce/幂等/重放防护                                  │
   └───────────────────────────────────────────────────────────────────────────────┘
```

| 组件 | 职责 | 是否共域 |
|------|------|:--:|
| 状态机引擎 | 驱动 §5 转换、超时、幂等 | 行为由规范定义（共域） |
| Exchange Store | 持久化交换/VDC/信誉 | 实现可私有，数据可导出 |
| 验收调度 | 按 rule 选验收器并记录证据 | 接口共域，验收器可竞争 |
| 结算调度 | 调结算适配器 lock/release/refund | 接口共域，适配器可换 |
| 信誉账本 | 签名哈希链、可复验 | 格式共域，数据可迁移 |
| 注册表服务 | 提供本体/规则 | **完全开放** |
| 身份/验签 | 验签、防重放 | 算法共域 |

## 4. 部署与联邦视图

```
   Node-0 (我们运营/参考)        Node-1 (某行业自建)        Node-N (某企业自建)
        │                            │                          │
        └──────── 同一套 L0–L3 规范 + 开放注册表 ────────────────┘
                         │
            陌生 Agent 跨节点交易：
            - VDC 由当事方签名，校验不依赖签发节点
            - 信誉条目可携带 + 跨节点复验（哈希链 + 验签）
            - 注册表（本体/规则）开放复制，多节点一致
```

**联邦原则**：
- 节点**无中心**；Node-0 只是流动性最大、最被信任的那个（靠服务质量，不靠垄断）。
- **凭证与信誉可携带**：Agent 换节点不丢历史；任何节点/任何人可复验他处签发的 VDC。
- **注册表开放复制**：本体/规则是公共数据，节点间保持版本一致（引用父规则 ID）。
- 这保证了底线第 4 条"无单一闸门、可退出"。

## 5. 端到端数据流（happy path）

```
A.manifest 发现 B ─► A 创建 Exchange(type,qty,rule,price,timeouts)
   └─ Node 校验 A 签名/nonce ─► 持久化 PROPOSED
A+B contract ─► CONTRACTED（条款定型，含验收 rule 与各超时）
A escrow ─► Settlement.lock() ─► ESCROWED（得 escrow_handle）
B 执行 ─► 产出 deliverable ─► 计算 result_hash ─► provider_sig
   └─ deliver ─► DELIVERED（VDC 半签：仅 provider_sig）
Node 验收调度 ─► Verifier.verify(deliverable, rule)
   ├─ pass ─► VERIFIED ─► A 复核 VDC ─► client_sig ─► confirm
   │           └─ Settlement.release() ─► SETTLED ✅ ─► 信誉 +1（签名链）
   └─ fail ─► Settlement.refund() ─► REJECTED ─► 信誉 记负
```

**数据落点与可复验性**：
- `deliverable` 原文存于 Node（或 A/B 本地），VDC 只存 `result_hash`（隐私 + 可校验）。
- **复验者**只需 `deliverable + rule(+schema) + VDC + 双方公钥`，即可：① 验签 ② 复跑验收 ③ 比对 result_hash ④ 校信誉链——**全程不需要信任 Node**。

## 6. 信任与安全架构（核心）

### 6.1 威胁模型

| 威胁 | 说明 | 对策（架构层） |
|------|------|----------------|
| **伪交割/空单** | 无真实交付却出凭证 | 双签 + result_hash + 证据等级；验收可被任何人复跑 |
| **重放** | 重复提交旧请求/旧 VDC | `nonce` 唯一 + `idempotency_key` + 重放窗口 |
| **篡改** | 改字段冒充 | Canonical JSON + Ed25519 签名覆盖全部业务字段 |
| **抵赖** | 事后否认签过 | 签名不可否认；信誉哈希链断链即标 UNVERIFIED |
| **恶意节点** | 运营方伪造/隐瞒 | **不信任节点**：一切可第三方复验；多节点联邦可绕开 |
| **恶意验收者** | evaluator 乱判（业界公认未解） | v0：确定性验收（可复跑，验收者无自由裁量）；未来：见证 + 质押罚没 + 多验收者 |
| **女巫** | 刷信誉 | v0 仅标注；未来：质押/成本门槛/见证加权 |
| **托管滥用** | 充值不释放/拒签 | 托管 + 超时自动退款 + 状态机强约束 |

### 6.2 关键安全机制

- **签名范围**：每个签名覆盖"排除 signatures 后的 canonical 字节"；provider_sig 覆盖交付声明，client_sig 覆盖确认（含对 provider_sig 的认可）。
- **防重放**：`(agent_id, nonce)` 全局唯一；`idempotency_key` 保证同操作幂等；超出重放窗口的请求拒绝。
- **崩溃恢复**：状态变更**先持久化意图、后执行副作用**（capture-before-execute）；重启时检测"孤儿意图"并补完（借鉴业界 Verity 做法）。详见《详细设计》。
- **密钥轮换**：agent 可发布密钥历史，旧签名按签发时有效性判定（v0 简化，接口预留）。
- **最小信任**：验收用确定性规则，**剥夺验收者的自由裁量**，从源头压制"恶意 evaluator"——这正是业界（ERC-8183 自承认）的痛点切入口。

### 6.3 "不被统治"由架构兜底

> 即使我方节点宕机/作恶，用户也能：① 用手里的 VDC 向任何人证明交割；② 在别的节点继续；③ 自托管。**这是开放与平等的硬保证，不靠承诺，靠可复验性 + 联邦 + 可导出。**

## 7. 中立性的架构保证（防止自己变成统治者）

| 风险（堕落方向） | 架构防线 |
|------------------|----------|
| 变成唯一闸门 | 多节点联邦 + 裸 HTTP 接入 + 数据可导出 |
| 私有化标准 | L0–L3 开放许可（CC-BY/Apache）+ 一致性测试套件 + 认证商标 |
| 绑死结算/链/币 | 结算外置为可换适配器；核心不持币、不强制介质 |
| 凭证封建主义 | 验收/见证为可竞争市场；信誉可携带可复验 |
| 黑箱 | 证据等级强制标注；一切可独立复算 |

## 8. 演进路线（架构层）

```
v0  软件 · 单笔 · 确定性验收 · Mock 结算 · 单/多节点      ← 参考实现已完成
v1  LLM 验收 · DID · 密钥轮换 · TS SDK · 争议槽           ← 骨架已落地（见详细设计 §14.18）
v2  见证/质押 · CBOR 双轨 · 信誉聚合/gate · 联邦 import   ← 特性开关 + 测试覆盖
v3  energy.*/actuation.* + ISO15118 模拟器               ← 可测会话，非真桩
```

> 未发布项：商标/对外 plugfest/一致性认证标；真实 x402/AP2/法币伙伴需外部集成。

> 每一步**只扩本体/旁路/接入面，不破坏 L0–L3 核心**——这是普适通用的架构纪律。
