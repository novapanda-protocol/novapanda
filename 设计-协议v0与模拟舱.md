# Troodon 协议 v0 · 开工级设计

> **依据**：《行动计划-从0到1》阶段 1（规范 v0）+ 阶段 2（模拟舱 demo）。
> **协议名**：Troodon（已定稿）。对外发布与商标待决策。
> **目标**：让两个**事先互不认识的陌生软件 Agent**，在无预建关系下完成一次**可验证交割**，模拟未来物理世界「跨主体、无预建关系」的交换。
> **铁律**：不绑币 · 结算无关 · 可复验 · 可退出。

---

## ★ 信仰（项目为何存在 · 设计的灵魂）

> **北极星**：让一切智能体、智能设备、大模型，都能在**无预建关系**下，用同一套**可验证的交割语言**彼此交换——真正实现「**智能万物交换**」。

我们相信：

1. **智能世界正在到来** —— 智能体与设备将成为交换与协作的**主体**，而不只是工具。
2. **价值的锚点正在迁移** —— 从「账户里的数字」迁到「网络上可验证的交付」：**先承认交付，再承认付款**。
3. **交换的语法应属于所有人** —— 像 TCP/IP、像 HTTP，是**公共的语法**，不属于任何一家公司、一条链、一种货币。
4. **开放与平等高于一切** —— 人人可**低成本接入**，不被大厂或大链统治；我们可以最强，但必须**可复验、可替代、可退出**。
5. **普适通用、降低门槛** —— 不绑垂直/框架/语言/货币；**绝不逼任何人改造自己来迁就标准**。
6. **永不铸币、永不收协议租金** —— 我们做交换的**能力与通道**，不做收银台，不做新的统治者。
7. **人类保留终审** —— 机器升到交换的密度，人退到宪法的高度；自由、生命、重大责任永远属于人。

> 一句话：**我们不是要造一种新的钱，而是要给智能世界一种「承认彼此付出」的共同语言——开放、普适、谁都能用、谁都拿不走。**

---

## ★ 设计底线（哲学不可妥协 · 凌驾于一切设计决策之上）

> 这是项目的**哲学地板**，是规范的最低线。任何字段、接口、实现，凡与之冲突，**一律以底线为准、推翻设计**。

1. **普适通用** — 不绑任何垂直/框架/语言/传输/链/货币。核心只描述「交付与交换」，与具体业务无关；**新领域靠扩资源本体，绝不改核心**。
2. **降低接入难度** — 多入口、低门槛（Skill 零代码 / MCP / A2A / 适配器 / SDK / 裸 HTTP）；**守规范即接入**；**绝不逼任何人改造自己来迁就标准**。
3. **要标准** — 一致性即成员资格；核心最小、可版本化、可认证（Conformance Suite + 认证商标）；像 HTTP 一样多实现、可互通。
4. **要开放** — 规范 CC-BY、代码 Apache、治理走向基金会；**不铸币、不抽协议租金、无单一闸门、可退出**；不被任何一家（含我们自己）统治。

**Litmus test（每个设计决策都要过的一问）**：

> 「一个我从没见过的、用**别的语言 / 别的框架 / 别的结算方式**的陌生 Agent，能不能在**不依赖我方任何专有组件、不经我方许可**的情况下，照规范完成一次交割，并被**任何人独立复验**？」
>
> 答案必须永远是 **能**。

## 0. 范围（先把边界钉死，防止零散扩张）

**v0 要做**：单笔、单任务类型、单规则的「陌生 Agent 交割」全流程 + 可复验信誉。

**v0 明确不做**（写下来就是为了不做）：
- 跨类折算（电↔推理）
- 多跳交换链
- 第三方见证 / 仲裁裁决（争议只留状态位，不实现解决）
- 任何上链 / 真实法币（结算用 mock，留适配口）
- 完整 DID（用 Ed25519 keypair 起步）
- 女巫攻击防御（仅标注风险）

**语言**：规范**语言中立**（JSON），任何语言可实现；**参考实现建议 Python 3.11+**（一个人写得快、加密/校验库齐全、demo 清晰）。若你更想要 TypeScript 也行，规范不受影响。

## 1. 角色与术语

| 角色 | 说明 |
|------|------|
| **Client（A，委托方）** | 发任务、出价、锁托管、最终确认签字 |
| **Provider（B，执行方）** | 接任务、交付结果、交付签字 |
| **Exchange Node（交换节点）** | 参考服务：撮合、托管、验收、签发 VDC、写信誉。**你运营的「零号节点」** |
| **Verifier（验收器）** | 按规则判定交付是否成立。v0 = 确定性校验（schema/哈希） |
| **Settlement Adapter（结算适配器）** | 可插拔的「结算腿」。v0 = mock；未来 = 法币持牌伙伴。**不绑币** |
| **Reputation Log（信誉账本）** | 只追加、签名、哈希链；任何人可复验 |

| 术语 | 含义 |
|------|------|
| **VDC** | Verifiable Delivery Claim，可验证交割凭证（核心产物） |
| **Exchange** | 一次交换的完整生命周期对象（携带状态机） |
| **Rule** | 验收标准（规则 ID 引用，不写具体价钱） |

## 2. 资源本体 v0（Resource Ontology）

**类型码格式**：`domain.category.subtype`（点分、带命名空间、可版本化）。

**v0 实现（软件类）**：

```
compute.inference.tokens      # 推理 token 用量
data.extraction.structured    # 结构化抽取（首个 demo 用）
service.task.generic          # 通用任务
```

**保留、暂不实现（物理类，脊椎预留）**：

```
energy.electric.dc            # 预留
actuation.robot.grasp         # 预留
```

每个类型定义：`unit`（计量单位）、`quantity_semantics`（数量含义）、`required_evidence`（必需证据字段）。注册表为 JSON，带 `version`，新增类型须登记。

**示例（本体条目）**：

```json
{
  "type": "data.extraction.structured",
  "version": "0.1",
  "unit": "document",
  "quantity_semantics": "处理的文档数",
  "required_evidence": ["result_hash", "schema_id", "started_at", "finished_at"]
}
```

## 3. 规则注册表 v0（Rule Registry）

**Rule = 验收标准的引用**。规则 ID 进凭证，具体校验逻辑在注册表。

**v0 规则**：

```json
{
  "rule_id": "R-extract-schema-v1",
  "version": "0.1",
  "acceptance": "deterministic",
  "check": {
    "kind": "json_schema_plus_hash",
    "schema_ref": "schemas/invoice_extract.schema.json",
    "require_hash_match": false
  },
  "description": "输出须通过指定 JSON Schema 校验；可选校验结果哈希。"
}
```

> 选确定性校验是因为它**无歧义、可被任何人复验**——这是 v0 最干净的「先证交付」。

## 4. VDC v0 · 字段定义

**规范 JSON 结构**（签名字段在签名时排除）：

| 字段 | 类型 | 必填 | 含义 |
|------|------|:--:|------|
| `vdc_version` | string | ✓ | 如 `"0.1"` |
| `vdc_id` | string | ✓ | 全局唯一（ULID/uuid） |
| `parties.client` | string | ✓ | A 的公钥指纹（agent id） |
| `parties.provider` | string | ✓ | B 的公钥指纹 |
| `resource_type` | string | ✓ | 引用本体类型码 |
| `quantity` | number | ✓ | 数量 |
| `result_hash` | string | ✓ | 交付结果的 SHA-256 |
| `rule_id` | string | ✓ | 引用规则注册表 |
| `evidence.level` | enum | ✓ | `self_reported`/`dual_signed`/`metered`/`third_party_witnessed` |
| `evidence.started_at` | string(ISO) | ✓ | 交付起 |
| `evidence.finished_at` | string(ISO) | ✓ | 交付止 |
| `evidence.metering` | object | ✗ | 计量明细（可选） |
| `state` | enum | ✓ | 见 §5 状态机 |
| `idempotency_key` | string | ✓ | 幂等键，防重放 |
| `nonce` | string | ✓ | 随机数，防重放 |
| `prev_hash` | string | ✗ | 信誉链上一条哈希（GENESIS 为首条） |
| `created_at` | string(ISO) | ✓ | 创建时间 |
| `signatures.provider_sig` | string | ✓ | B 交付签（Ed25519, base64url） |
| `signatures.client_sig` | string | 终态✓ | A 确认签 |

**规范化与签名规则**：
1. **Canonical JSON** = 键按字典序排序、UTF-8、无多余空白。
2. 签名对象 = 排除 `signatures` 后的 canonical 字节。
3. **哈希** = SHA-256(canonical bytes)；**签名** = Ed25519。
4. 「**有效交割 VDC**」= `state == SETTLED` 且 `provider_sig` 与 `client_sig` 均验签通过。

**示例**：

```json
{
  "vdc_version": "0.1",
  "vdc_id": "01J9X8...",
  "parties": { "client": "ed25519:A1b2...", "provider": "ed25519:C3d4..." },
  "resource_type": "data.extraction.structured",
  "quantity": 1,
  "result_hash": "sha256:9f86d0...",
  "rule_id": "R-extract-schema-v1",
  "evidence": { "level": "dual_signed", "started_at": "2026-06-28T03:00:00Z", "finished_at": "2026-06-28T03:00:04Z" },
  "state": "SETTLED",
  "idempotency_key": "task-7741",
  "nonce": "b7e1...",
  "prev_hash": "GENESIS",
  "created_at": "2026-06-28T03:00:05Z",
  "signatures": { "provider_sig": "base64url...", "client_sig": "base64url..." }
}
```

## 5. 交换状态机

```
PROPOSED ──contract──► CONTRACTED ──escrow──► ESCROWED ──deliver──► DELIVERED ──verify(pass)──► VERIFIED ──confirm──► SETTLED ✅
   │                        │                     │                                    │
   │ client_cancel          │ timeout             │ timeout/verify(fail)               │ (无)
   ▼                        ▼                     ▼                                    ▼
 CANCELLED              EXPIRED_REFUNDED      REJECTED / EXPIRED_REFUNDED          (终态)

任何 DELIVERED 后争议 → DISPUTED（仅占位，不实现解决）
```

| 转换 | 触发者 | 前置 | 效果 |
|------|--------|------|------|
| `contract` | A+B | 双方就 type/qty/rule/price/timeouts 达成 | 进 CONTRACTED |
| `escrow` | A | settlement.lock 成功 | 进 ESCROWED，得 escrow_handle |
| `deliver` | B | 提交结果 + result_hash + provider_sig | 进 DELIVERED |
| `verify` | Node/Verifier | 跑 rule | pass→VERIFIED；fail→REJECTED(退款) |
| `confirm` | A | 校验 VDC，client_sig | 进 SETTLED，settlement.release |
| `timeout` | Node | 超过各阶段时限 | EXPIRED_REFUNDED |
| `client_cancel` | A | 仅 PROPOSED/CONTRACTED | CANCELLED |

**超时**：`escrow_timeout` / `deliver_timeout` / `verify_timeout`（合约里约定）。
**幂等/防重放**：所有写操作带 `idempotency_key`；同键返回既有结果；`nonce` 防 VDC 重放。

## 6. 协议接口（v0 用 HTTP/JSON）

Exchange Node 暴露一个 `Exchange` 资源的生命周期：

```
POST /exchanges                 # A 发起：声明 type/qty/rule/price/timeouts/provider → PROPOSED
POST /exchanges/{id}/contract   # 双方确认条款 → CONTRACTED
POST /exchanges/{id}/escrow     # A 锁托管 → ESCROWED
POST /exchanges/{id}/deliver    # B 提交结果+签名 → DELIVERED
POST /exchanges/{id}/verify     # 触发验收（也可自动） → VERIFIED/REJECTED
POST /exchanges/{id}/confirm    # A 确认签字 → SETTLED
GET  /exchanges/{id}            # 查状态与 VDC
GET  /vdc/{vdc_id}              # 取 VDC
GET  /reputation/{agent_id}     # 取信誉记录（可复验）
GET  /registry/types            # 资源本体
GET  /registry/rules            # 规则注册表
```

所有请求/响应体引用 §4 的 VDC 字段与 §5 状态。

## 7. 结算适配器接口（不绑币、可插拔）

```python
class SettlementAdapter(Protocol):
    def lock(self, amount: Decimal, currency: str, ref: str) -> EscrowHandle: ...
    def release(self, handle: EscrowHandle) -> SettlementReceipt: ...
    def refund(self, handle: EscrowHandle) -> SettlementReceipt: ...
```

- **v0 实现**：`MockSettlement`（内存账本，打印流水）。
- **未来**：`FiatAdapter`（对接持牌支付/清算伙伴）；**自己不持牌**。
- 关键：结算是**独立、可换**的一条腿，**VDC 不依赖它**——这就是「结算无关」。

## 8. 验收器接口

```python
class Verifier(Protocol):
    def verify(self, deliverable: bytes, rule: Rule) -> VerifyResult:
        # VerifyResult = {passed: bool, evidence_ref: str, reason: str}
        ...
```

- **v0 实现**：`SchemaVerifier`（确定性）：用 `jsonschema` 校验结构 +（可选）比对 `result_hash`。**任何人拿同样输入可复算同样结论**。
- **未来变体**：`LLMJudgeVerifier`（质量 SLA，证明能泛化到模糊任务）——v0 不做。

## 9. 信誉账本（可复验）

每个**终态** Exchange → 一条**签名、哈希链**的信誉条目：

```json
{
  "entry_id": "01J9...",
  "agent_id": "ed25519:C3d4...",
  "vdc_id": "01J9X8...",
  "outcome": "SETTLED",          // 或 REJECTED / EXPIRED_REFUNDED
  "resource_type": "data.extraction.structured",
  "rule_id": "R-extract-schema-v1",
  "prev_hash": "sha256:...",     // 链接上一条；首条 GENESIS
  "created_at": "...",
  "node_sig": "base64url..."
}
```

- 只追加；`prev_hash` 断链即标 `UNVERIFIED`（借鉴业界做法）。
- **任何人可独立复算链 + 验签**——不是「相信我」。
- ⚠️ **女巫攻击**（刷信誉）v0 不防，仅标注为已知风险。

## 12. 模拟舱 Demo（阶段 2）

**剧本**：Agent A 需要从一张发票文档里抽取结构化字段；Agent B 提供 `data.extraction.structured`。**A 和 B 事先互不认识，只知道对方公钥。**

**跑通完整仪式（happy path）**：

```
1. A 发起 Exchange：type=data.extraction.structured, qty=1, rule=R-extract-schema-v1,
   price=0.01（mock 单位）, timeouts=...  → PROPOSED
2. A、B contract 确认条款                                   → CONTRACTED
3. A escrow 锁 0.01（MockSettlement.lock）                  → ESCROWED
4. B 抽取 → 产出 JSON + result_hash + provider_sig，deliver → DELIVERED
5. SchemaVerifier 校验通过                                  → VERIFIED
6. A 校验 VDC、client_sig，confirm（release 0.01 给 B）      → SETTLED ✅
7. 信誉账本写入 B 的 +1 条目
8. demo 打印整张签名 VDC + 信誉条目
9. reverify.py 独立复算：验签 + 复跑 schema 校验 + 校信誉链 → 全绿
```

**另外演两条分支**（证明健壮）：
- **超时退款**：B 不交付 → 到 `deliver_timeout` → EXPIRED_REFUNDED，A 拿回 0.01。
- **验收拒绝**：B 交付不合 schema → REJECTED，退款，B 信誉记一条负面。

**演示价值**：两个陌生 Agent，无预建关系，靠「可验证交付 + 双签 + 可复验信誉」完成交换，结算腿可换、不绑币——直击命题。

## 14. 泛化与互操作（一等设计目标）

> 两个 Agent 的 demo 只是**最小证明**；架构从第一天起就按「**任意异构 Agent（行业智能体、OpenClaw 等引擎、企业/品牌 Agent）都能快速对接**」设计。这是「标准」与「一个产品」的根本区别。

### 14.1 原则：一致性即接入（conformance = membership）

- 接入的**唯一门槛** = 能产出/消费**合规 VDC** + 按状态机在约定传输上交互。**像 HTTP**：守规范即互通。
- **不需要用我们的 SDK、不需许可、不绑语言/框架**。
- **核心规范保持最小、框架无关**；框架相关的东西全放「适配器」，绝不进核心——这是泛化的根本保证。
- Exchange Node **可多实例、人人可自建**（联邦）；凭证/信誉可跨节点复验。

### 14.2 能力清单与发现（Agent Manifest）

任何 Agent 暴露 `/.well-known/troodon.json`，声明能力 → **陌生行业 Agent 无预建关系即可发现并成交**（「跨主体」泛化的关键）：

```json
{
  "agent_id": "ed25519:...",
  "capabilities": [
    {"resource_type": "data.extraction.structured", "rules": ["R-extract-schema-v1"], "price": {"amount": 0.01, "currency": "mock"}}
  ],
  "endpoints": {"exchange": "https://...", "transport": ["http", "a2a", "mcp"]}
}
```

### 14.3 传输无关 + 桥接现有栈（互操作不替代）

- 核心**传输无关**；提供绑定：HTTP（v0）、**A2A**、**MCP**。OpenClaw 等已讲 MCP/A2A 的 Agent **近乎零改造接入**。
- **x402 / AP2 作为结算适配器**（§7 的一种实现）：需兑现时调它们 → 我们在上、它们在下。
- 物理侧未来同理：ISO 15118 等做成适配器接入。

### 14.4 接入面总览（多入口，门槛越低越接近「万物交换」）

接入面越多，能插进来的智能体/设备越多——这正是通往**万物交换**的路。**不逼任何人改造自己，给每一类 Agent 一条几乎零成本的入口。** 所有入口最终都映射到同一套 §4 VDC + §5 状态机：

| 入口 | 适用对象 | 门槛 | 说明 |
|------|----------|:--:|------|
| **Skill** | 支持技能的 Agent（Claude、Cursor agents 等） | 最低 | 装上即学会发任务/托管/交付/签 VDC/查信誉，**零代码**。可能是主入口 |
| **MCP 绑定** | 已讲 MCP 的 Agent / 工具 | 极低 | 近乎零改造接入 |
| **A2A 绑定** | 已讲 A2A 的 Agent | 极低 | 直接对端互通 |
| **适配器** | OpenClaw / LangChain 等框架 | 低 | 薄封装把能力包成 Provider/Client，**核心不动** |
| **SDK** | 自建 Agent / 深度集成 | 中 | 原生调用，最灵活（`sdk/` 客户端库） |
| **裸 HTTP/JSON** | 任意语言/任意系统 | — | 直接按规范实现，**无需任何我方组件**（像 HTTP） |

代码占位：

```
sdk/                    # 原生 SDK（client.py / vdc.py / identity.py）
adapters/openclaw.py    adapters/mcp.py    adapters/a2a.py    adapters/langchain.py
skill/                  # 可安装的 Troodon Skill（封装整套交换仪式）
```

> 适配器/Skill/绑定只做「翻译」：外部调用 ↔ Troodon Exchange 生命周期。**核心规范永远不动**——这是泛化的根本保证。

### 14.5 一致性测试套件（「快速对拉」的钥匙）

提供可下载的 **Conformance Test Suite**：任何 Agent/平台跑一遍即可**自助认证「Troodon 兼容」**。
- 这是让「快速通过标准对接」可操作的东西。
- **绑认证商标**（见《投资战略》§7.1 商标层）：测过才能打「兼容」标 → 既保证互通质量，又不靠垄断。

### 14.6 异构 demo（证明靠规范、不靠 SDK）

模拟舱两个 Agent **故意用不同栈**（如：一个纯 Python 直连 HTTP，一个走 MCP 适配器模拟 OpenClaw 式 Agent），且只知对方公钥 → 证明互通来自**遵守规范**，而非「都用同一个 SDK」。
