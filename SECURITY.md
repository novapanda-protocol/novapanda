# SECURITY · 安全与威胁模型（摘要）

> **状态**：现行摘要 · 2026-07-08  
> **范围**：协议与参考实现的**公开**威胁模型；运营细节可另有内部 runbook。  
> **配套**：[CHARTER.md](CHARTER.md) · [spec/SPEC.md](spec/SPEC.md)

---

## 1. 安全目标

| 目标 | 含义 |
|------|------|
| **真实性** | VDC 与关键报文不可伪造（Ed25519） |
| **完整性** | canonical + 哈希绑定交付与条款 |
| **复验独立性** | 不信任单一节点即可验真 |
| **抗重放** | nonce / 幂等键 |
| **密钥隔离** | Agent 私钥永不上送节点 |
| **可退出** | 不因锁死节点而丧失证据 |

非目标：协议不提供银行级资金保管；真钱安全在结算伙伴。

---

## 2. 资产

- Agent 私钥与会话 nonce  
- VDC 与 deliverable 原文（后者常需授权）  
- Exchange 条款与 `terms_hash`  
- 信誉链、Manifest  
- Operator Session / 运营 token（身体，非协议核心）  
- 结算轨凭证与 Claim（旁路）

## 3. 对手与主要威胁

| ID | 威胁 | 缓解 |
|----|------|------|
| T1 | 伪造 Agent 身份 | 公钥即身份；无私钥不可签 |
| T2 | 篡改 deliverable 仍骗 SETTLED | 内容哈希进 VDC；规则可复算 |
| T3 | 节点谎报状态 | 当事方签名与独立 reverify；节点可替代 |
| T4 | 重放 propose/ack | nonce 唯一 + 时间窗；幂等键 |
| T5 | Manifest 夸大能力 | **必须验签**后再信 |
| T6 | 恶意验收规则歧义 | 确定性 verifier；LLM judge 走 audit 快照 |
| T7 | 信誉刷单 / 诽谤 | 链可复验；可选见证与 stake |
| T8 | 结算适配器双花/崩溃 | capture-before-execute；recover() |
| T9 | Claim 无锚发行 / 双花 | 要求 VDC 锚；轨侧原子 redeem（Profile） |
| T10 | 运营账户接管 | OTP/Session 安全；**禁止**替代 Agent 签 |
| T11 | 拒绝服务刷写 API | 配额、限速、策略 A/B（身体） |
| T12 | 供应链（依赖投毒） | 锁版本、审发布物、最小权限 CI |
| T13 | 隐私过度暴露 | 链上优先哈希；原文出带授权 |
| **T14** | **交割未达标却轨上放款 / 假 SETTLED** | 见内部《交换结算清算-价值闭环与安全》E2E-01/02；NP-SETTLE A5 |
| **T15** | **mock 或静默换轨骗资** | NP-SETTLE A6；UC-45；表述红线 |

端到端（交换↔结算↔清算边界）控制清单与价值闭环：内部设计 `交换结算清算-价值闭环与安全.md`。  
公开结算适配规范：[`profiles/NP-SETTLE.md`](profiles/NP-SETTLE.md)。

## 4. 密钥与角色

| 密钥 | 持有 | 禁止 |
|------|------|------|
| Agent Ed25519 | 仅用户/设备本地 | 上送节点「代管方便试用」 |
| 节点信誉/服务键 | 运维 HSM 或受限主机 | 写进前端 |
| Operator 密码/会话 | IdP 最佳实践 | 用于 exchange 鉴 |

轮换：Agent 换钥 = 新 `agent_id`；旧 VDC 仍绑旧钥可验。节点键轮换不得改写历史 entry 的可验性（新签名仅用于新条目）。

## 6. 披露（Security Disclosure）

### 6.1 范围

欢迎报告：伪交割、签名/canonical 缺陷、节点越权、结算适配双花或假 SETTLED、供应链投毒线索等。  
**不要**把 Trial mock 上的「没到真钱」当成漏洞。

### 6.2 如何报告

1. **优先**：GitHub **Private Security Advisory**（若已启用），或维护者私信。  
2. **邮箱（Steward 过渡期）**：在仓库 About / 组织简介中公布的安全联系；若暂缺，使用公开 issue **仅**声明「有安全问题求私信渠道」，**不要**贴利用细节。  
3. 给出：影响版本/提交、复现步骤、是否已公开、你的联系方式。

### 6.3 SLA（目标 · Steward 阶段尽力而为）

| 步骤 | 目标时间 |
|------|----------|
| 确认收到 | **3 个工作日**内 |
| 初步严重级别评估 | **7 个工作日**内 |
| 修复或缓解方案（高危） | **90 日**内尽力；复杂密码学/生态依赖可延长并沟通 |
| 协调披露 | 与报告人商定；默认可修复后公开致谢 |

无法承诺法定「漏洞赏金」；致谢名单 MAY 维护。

### 6.4 严禁

- 公开完整 0-day 利用换取关注  
- 对生产真钱轨做未授权渗透（找伙伴的责任披露渠道）  
- 要求「付封口费」

实现者核对清单见上文 §5；端到端结算威胁见内部闭环文与本文件 T14/T15。

## 7. 修订

威胁模型随 Profile（物理、Claim 转让）扩展而更新；破坏性安全假设变更走 VERSIONING Breaking 级讨论。  
披露 SLA 可在基金会阶段收紧并公示。
