## 1. 数据模型（权威定义）

所有对象序列化为 **Canonical JSON**（见 §2）。下表 `R`=必填，`O`=可选。

### 1.1 AgentIdentity / agent_id
- `agent_id`：字符串，格式 `ed25519:<base58(pubkey-32B)>`。
- 密钥：Ed25519；私钥本地保管，永不上传。

### 1.2 AgentManifest（`/.well-known/troodon.json`）
| 字段 | 类型 | R/O |
|------|------|:--:|
| `agent_id` | string | R |
| `capabilities[]` | array | R |
| `capabilities[].resource_type` | string | R |
| `capabilities[].rules[]` | string[] | R |
| `capabilities[].price` | {amount:number, currency:string} | O |
| `endpoints.exchange` | url | R |
| `endpoints.transport[]` | enum[] (`http`/`a2a`/`mcp`/`skill`) | R |
| `pubkey` | string(base64url) | R |
| `sig` | string | R（自签名 manifest） |

### 1.3 OntologyType（资源本体条目）
| 字段 | 类型 | R/O |
|------|------|:--:|
| `type` | string（`domain.category.subtype`） | R |
| `version` | string | R |
| `unit` | string | R |
| `quantity_semantics` | string | R |
| `required_evidence[]` | string[] | R |

### 1.4 Rule（规则注册表条目）
| 字段 | 类型 | R/O |
|------|------|:--:|
| `rule_id` | string | R |
| `version` | string | R |
| `acceptance` | enum(`deterministic`/`attested`/`judged`) | R |
| `check` | object（见下） | R |
| `parent_rule_id` | string | O（跨域/继承时引用父规则） |
| `description` | string | R |

`check`（v0 `json_schema_plus_hash`）：`{kind, schema_ref, require_hash_match:bool}`。

### 1.5 Exchange（交换生命周期对象）
| 字段 | 类型 | R/O | 说明 |
|------|------|:--:|------|
| `exchange_id` | string(ULID) | R | |
| `client` / `provider` | agent_id | R | |
| `resource_type` | string | R | 引用本体 |
| `quantity` | number | R | |
| `rule_id` | string | R | |
| `price` | {amount,currency} | R | currency 可为 `mock` |
| `timeouts` | {escrow,deliver,verify}（秒） | R | |
| `state` | enum | R | 见 §3 |
| `escrow_handle` | string | O | ESCROWED 后 |
| `deliverable_ref` | string | O | 存储引用 |
| `vdc_id` | string | O | 生成后 |
| `idempotency_key` | string | R | |
| `created_at`/`updated_at` | ISO8601 | R | |

### 1.6 VDC（可验证交割凭证）
权威字段见《设计》§4。补充约束：
- `vdc_id` = ULID；`result_hash` = `sha256:<hex>`；`evidence.level` ∈ {`self_reported`,`dual_signed`,`metered`,`third_party_witnessed`}。
- `state` 只取终态相关值（`SETTLED`/`REJECTED`/`EXPIRED_REFUNDED`）。
- `signatures.provider_sig` R；`signatures.client_sig` 在 `SETTLED` 时 R。

### 1.7 VerifyResult
`{passed:bool, evidence_ref:string, reason:string, verifier_id:string, replay_inputs_ref:string}`
- `replay_inputs_ref`：指向复跑所需输入（deliverable + rule + schema）的引用，**保证可复验**。

### 1.8 ReputationEntry
| 字段 | 类型 |
|------|------|
| `entry_id` | ULID |
| `agent_id` | agent_id |
| `vdc_id` | string |
| `outcome` | enum(`SETTLED`/`REJECTED`/`EXPIRED_REFUNDED`) |
| `resource_type`/`rule_id` | string |
| `seq` | int（该 agent 链内序号） |
| `prev_hash` | `sha256:` 或 `GENESIS` |
| `created_at` | ISO8601 |
| `node_sig` | string |

### 1.9 ErrorResponse
`{error_code:string, message:string, exchange_id?:string, retriable:bool}`

## 2. 规范化与密码学（Canonicalization & Crypto）

| 项 | 规定 |
|----|------|
| 序列化 | **Canonical JSON**：键按 UTF-8 码点字典序排序；无多余空白；数字最简形式；字符串 NFC。（等价 RFC 8785 JCS） |
| 哈希 | SHA-256；表示为 `sha256:<lowercase-hex>` |
| 签名 | Ed25519；输出 base64url(无填充) |
| 签名输入 | 对象去掉 `signatures`（或对应签名字段）后的 canonical 字节 |
| 公钥/ID | `ed25519:` + base58(pubkey) |
| 结果哈希 | `result_hash = sha256(canonical(deliverable))`；deliverable 为 JSON 时先 canonical |

**签名范围（明确）**：
- `provider_sig` = Ed25519(provider_sk, canonical(VDC \ {signatures}))
- `client_sig`  = Ed25519(client_sk,  canonical(VDC \ {signatures.client_sig}))（即客户确认时已包含 provider_sig，表示"认可 B 的交付声明"）

## 3. 状态机详表

### 3.1 状态
`PROPOSED · CONTRACTED · ESCROWED · DELIVERED · VERIFIED · SETTLED(终) · REJECTED(终) · EXPIRED_REFUNDED(终) · CANCELLED(终) · DISPUTED(占位)`

### 3.2 转换表

| # | from → to | 触发者 | 守卫(guard) | 副作用 |
|---|-----------|--------|-------------|--------|
| 1 | PROPOSED→CONTRACTED | A+B | 条款一致、双方 ack | 冻结 type/qty/rule/price/timeouts |
| 2 | CONTRACTED→ESCROWED | A | Settlement.lock 成功 | 记 escrow_handle，启动 deliver_timer |
| 3 | ESCROWED→DELIVERED | B | provider_sig 验签通过、result_hash 一致 | 存 deliverable_ref，构造半签 VDC，启动 verify_timer |
| 4 | DELIVERED→VERIFIED | Node | Verifier.passed=true | 记 evidence_ref |
| 5 | VERIFIED→SETTLED | A | client_sig 验签通过 | Settlement.release，写 VDC(SETTLED)，写信誉 + |
| 6 | DELIVERED→REJECTED | Node | Verifier.passed=false | Settlement.refund，写 VDC(REJECTED)，写信誉 - |
| 7 | ESCROWED/DELIVERED→EXPIRED_REFUNDED | Node | 超 deliver/verify 时限 | Settlement.refund，写信誉(中性/-) |
| 8 | PROPOSED/CONTRACTED→CANCELLED | A | 尚未 escrow | 无资金动作 |
| 9 | DELIVERED→DISPUTED | A 或 B | 申诉（v0 仅置位，不解决） | 冻结，待人工/未来仲裁 |

### 3.3 不变量
- 资金只在 `escrow→release/refund` 流动，且每个 escrow_handle 恰好一次终结。
- 非法转换一律 `E_STATE_INVALID`。
- 所有写转换需带 `idempotency_key`；重复 → 返回既有结果（不重复副作用）。

## 4. API 契约（v0 HTTP/JSON）

**通用**：
- 请求头：`X-Agent-Id`、`X-Nonce`、`X-Signature`(对请求体 canonical 的 Ed25519)、`Idempotency-Key`。
- 验签失败→`401 E_SIG_INVALID`；nonce 重放→`409 E_REPLAY`。

| 端点 | 方法 | 请求体(要点) | 成功响应 | 主要错误码 |
|------|------|--------------|----------|-----------|
| `/exchanges` | POST | type,qty,rule_id,price,timeouts,provider | 201 Exchange(PROPOSED) | E_TYPE_UNKNOWN,E_RULE_UNKNOWN |
| `/exchanges/{id}/contract` | POST | ack + 条款摘要哈希 | 200 Exchange(CONTRACTED) | E_TERMS_MISMATCH,E_STATE_INVALID |
| `/exchanges/{id}/escrow` | POST | amount,currency | 200 Exchange(ESCROWED)+handle | E_ESCROW_FAILED |
| `/exchanges/{id}/deliver` | POST | deliverable, result_hash, provider_sig | 200 Exchange(DELIVERED)+半签VDC | E_HASH_MISMATCH,E_SIG_INVALID |
| `/exchanges/{id}/verify` | POST | (可自动触发) | 200 VERIFIED/REJECTED+VerifyResult | E_VERIFY_ERROR |
| `/exchanges/{id}/confirm` | POST | client_sig | 200 Exchange(SETTLED)+VDC | E_SIG_INVALID,E_STATE_INVALID |
| `/vdc/{vdc_id}` | GET | — | 200 VDC | E_NOT_FOUND |
| `/reputation/{agent_id}` | GET | ?since=seq | 200 ReputationEntry[] | — |
| `/registry/types` `/registry/rules` | GET | — | 200 列表 | — |

## 5. 错误模型

| error_code | 含义 | retriable |
|------------|------|:--:|
| `E_SIG_INVALID` | 签名验证失败 | 否 |
| `E_REPLAY` | nonce 重放/超窗 | 否 |
| `E_STATE_INVALID` | 非法状态转换 | 否 |
| `E_TERMS_MISMATCH` | 条款摘要不一致 | 否 |
| `E_HASH_MISMATCH` | result_hash 与 deliverable 不符 | 否 |
| `E_TYPE_UNKNOWN`/`E_RULE_UNKNOWN` | 本体/规则未注册 | 否 |
| `E_ESCROW_FAILED` | 结算锁定失败 | 是 |
| `E_VERIFY_ERROR` | 验收器异常（非 fail） | 是 |
| `E_TIMEOUT` | 阶段超时 | 否 |
| `E_NOT_FOUND` | 资源不存在 | 否 |

> 区分 **验收 fail（REJECTED，业务结果）** 与 **验收 error（E_VERIFY_ERROR，技术故障）**——前者退款入终态，后者可重试。

## 6. 验收契约（可复验是硬要求）

```python
class Verifier(Protocol):
    def verify(self, deliverable: bytes, rule: Rule) -> VerifyResult: ...
```
- **确定性要求**：相同 `(deliverable, rule, schema)` 必产生相同 `passed`。
- **第三方复跑**：`VerifyResult.replay_inputs_ref` 提供 deliverable+rule+schema 的获取方式；任何人可独立复跑得到同一结论——**验收不靠信任 Node**。
- v0 `SchemaVerifier`：`jsonschema` 校验 +（可选）`result_hash` 比对。
- 未来 `LLMJudgeVerifier`：多裁判取共识（未实现）；v1 已实现 schema 预检链 + judge_registry + HTTP/OpenAI 网关 + `llm_audit` 快照。

## 7. 信誉链（可复验）

- 每个 agent 一条**只追加哈希链**：`entry.hash = sha256(canonical(entry \ {node_sig}))`；`prev_hash` 链接前条，首条 `GENESIS`。
- `node_sig` = 节点对条目签名（标明出处）。
- **复验**：拉取 `/reputation/{agent}`，逐条验 `prev_hash` 链 + 验 `node_sig` + 比对引用的 VDC；断链 → 该点后全部标 `UNVERIFIED`。
- **跨节点**：信誉可携带；他节点复验只需公钥与链数据，**无需信任签发节点**。
- ⚠️ 女巫防御 v0 不做（标注），未来靠质押/见证加权。

## 8. 接入面契约（统一映射到 Exchange 生命周期）

| 接入面 | 映射方式 | 关键约定 |
|--------|----------|----------|
| **裸 HTTP** | 直接调 §4 端点 | 自行做 canonical + 签名 |
| **SDK** | `client.create/contract/escrow/deliver/confirm()` | 内部封装签名/nonce/幂等 |
| **MCP** | 把端点暴露为 MCP tools（`troodon.create_exchange`…） | tool 入参=端点入参；tool 内完成签名 |
| **A2A** | 把交换消息映射为 A2A 消息类型 | 对端互通靠 VDC，不靠 A2A 私有字段 |
| **Skill** | 技能封装"发任务/托管/交付/签VDC/查信誉"指令 | 内置 SDK 调用；零代码 |
| **适配器** | OpenClaw/LangChain：能力↔Provider/Client | 仅翻译，不改核心 |

> 所有接入面**最终产物相同**：合规 VDC + 信誉条目。任一接入面产出的 VDC 必须能被其他接入面/裸 HTTP 复验。

## 10. 持久化与并发

- **存储（v0）**：SQLite。表：`exchanges`、`vdcs`、`reputation_entries`、`nonces`、`idempotency`、`registry_types`、`registry_rules`。
- **并发**：对 `Exchange.state` 用**乐观锁**（version 字段 / `WHERE state=expected`）；并发转换只成功一个，另一个得 `E_STATE_INVALID`。
- **幂等**：`idempotency(key, request_hash) → response`；命中直接返回既有响应。
- **重放窗口**：`nonces(agent_id, nonce, seen_at)` 唯一约束；清理早于窗口（如 24h）的记录。

## 11. 崩溃恢复（capture-before-execute）

每个带副作用的转换（escrow/release/refund/写VDC/写信誉）：
1. **先持久化 INTENT**（`intent` 表：transition、exchange_id、payload、status=PENDING）。
2. 执行副作用（调结算/写记录）。
3. 标 INTENT=DONE。

重启时扫 `intent.status=PENDING`：
- 若副作用幂等（带 ref），**重放补完**；
- 标记完成。

→ 保证「没有决策会丢失」，即使在 release/refund 中途宕机也能恢复一致。

## 14. v0 实现增补（与代码对齐，2026-06-28）

以下为参考实现（Python，包名占位 `troodon`）落地后对本设计的澄清与增补，凡有出入以本节为准。

### 14.1 签名范围：`state` 不进签名
`state` 是生命周期标记，会随 `DELIVERED→SETTLED` 变化；若纳入签名，状态一变 provider_sig 即失效。故：
- `provider_sig = Ed25519(canonical(VDC 去掉 signatures、state))`
- `client_sig   = Ed25519(canonical(VDC 去掉 state、signatures.client_sig))`
代码：`troodon/vdc.py`（`_VOLATILE_FIELDS=("state",)`）。规范见 `spec/SPEC.md §2.1`。

### 14.2 分阶段超时（仅覆盖交付前 + 可选 verify）
`Exchange.timeouts` 按阶段（`contract`/`escrow`/`deliver`/`verify`）设 deadline。
交付前阶段超时 → `EXPIRED_REFUNDED`；若设 `verify` 超时，DELIVERED 后 client 未验收也会过期退款。
`engine.sweep(now_ts=…)` + 节点 `POST /admin/sweep`。环境变量 `TROODON_TIMEOUT_VERIFY`。

### 14.3 接入面（统一注册表）
所有接入面共享 `troodon/surfaces/base.OPERATIONS` 单一真相，仅做翻译，不改语义：
SDK / `MCPBinding` / `A2ABinding` / `ExchangeSkill` / `ProviderAdapter`（外部系统一个 `work_fn` 即可作为 provider 接入）。
一致性测试证明：同一笔交割经任一接入面，产出 VDC 等价且可独立复验。

### 14.4 请求级鉴权
`troodon/auth.py` 定义跨语言一致的请求签名：`canonical({method,path,nonce,body_sha256})`。
节点 `create_app(auth=True)` 启用：缺头/验签失败→401，nonce 重放→401，越权→403。
授权矩阵：propose=client；escrow/confirm=client；deliver=provider；contract/verify/cancel/expire/dispute/resolve=相关方。
SDK 每个 POST 自动带 `X-Agent-Id/X-Nonce/X-Signature`。GET 端点公开。

### 14.5 争议（DISPUTED）
`VERIFIED→DISPUTED→{SETTLED|REJECTED}`。`engine.dispute(by,reason)` / `engine.resolve(outcome,resolver)`；
节点 `POST …/dispute`、`POST …/resolve`。v0 由相关方裁决（生产应换独立仲裁者）；
裁决 settle 的依据是仲裁权威而非双签，故该路径 VDC 可能仅 provider 单签。

### 14.6 持久化与乐观锁
`troodon/store.py`：`InMemoryStore`（默认）/ `SQLiteStore`（落盘）。每个对象记 `_pv`（已持久化 version），
`save()` 以 `WHERE version=_pv` 做乐观锁，冲突抛 `ConcurrencyError`；支持同一操作内多次落盘（capture-before-execute 需要）。

### 14.7 结算 capture-before-execute + 恢复
结算适配器**幂等**（重放安全）。`confirm`/退款在调用外部结算前先落盘 `settlement_intent=pending`（`_capture_intent`），
完成后置 `done`。`engine.recover()` 扫描 `pending` 意图，幂等重放并补齐终态——保证「钱动了但状态没落盘」可恢复。

### 14.8 结算适配器
`SettlementAdapter` 抽象 + `MockSettlement`（内存幂等）。可插拔轨道：`GatewaySettlement(gateway)` 基类，
`X402Settlement`/`AP2Settlement` 为薄子类（escrow≈authorize、settle≈capture、refund≈void），`FakeGateway` 供测试。
换 rail 只需换 gateway——协议对资金不可知。

### 14.9 信誉持久化
`ReputationLog(node_identity, store=…)`：`InMemoryReputationStore` / `SQLiteReputationStore`。
哈希链跨进程延续、可独立复验（重算 `entry_hash`+链接+节点签名）。

### 14.11 生产硬化（2026-06-28 增补）

- 节点 `create_app(auth=True)` **默认开启鉴权**；测试/demo 可显式 `auth=False`。
- 启动 lifespan 自动 `engine.recover()`。
- nonce 持久化：`SQLiteStore.nonce_check_and_add()`，24h 窗口；重放 → HTTP 409 `E_REPLAY`。
- 错误码与 `spec/SPEC.md §13` 对齐（`E_SIG_INVALID` / `E_REPLAY` 等）。

### 14.12 Contract 双签 + Agent Manifest

- `terms_hash` 在 `propose` 时冻结；`POST …/contract` body 含对 `{action,exchange_id,terms_hash}` 的签名。
- client **与** provider 各 ack 一次后才 `CONTRACTED`；无鉴权时从签名反推 party。
- Agent manifest：`troodon/manifest.py` 的 `build_agent_manifest` / `verify_agent_manifest`；
  对端验签后可发现 `capabilities` 并发起交换（一致性 C7）。

### 14.13 开放注册表 + 部署配置 + x402 HTTP（2026-06-28）

- **注册表文件**：`spec/ontology.v0.json`（含 `energy.*` / `actuation.*` 预留）、`spec/rules.v0.json`；
  节点 `seed=True` 时默认加载；`GET /registry/types|rules` 只读发布。
- **可复验增强**：交付后设 `deliverable_ref=inline:{exchange_id}`；`VerifyResult.replay_inputs_ref`
  含 `rule_id` + `schema_inline`，第三方可独立复跑 Schema 验收。
- **部署**：`troodon/config.py` — `NodeConfig.from_env()`；
  `TROODON_AUTH` / `TROODON_DB` / `TROODON_SETTLEMENT` / `TROODON_X402_URL` 等。
  生产入口：`uvicorn troodon.node.app:create_app_from_config --factory`。
- **x402 HTTP**：`HttpX402Gateway` + `create_x402_fake_app()` 测试用 fake server；真实部署指向外部 402 网关 URL。

### 14.14 Deliverable 外存 + 仲裁 + AP2 + 复验增强（2026-06-28）

- **BlobStore**：`troodon/blobs.py` — `InMemoryBlobStore` / `SQLiteBlobStore`（`blobs` 表）；
  配置 `TROODON_DB` 时自动启用；Exchange JSON 只存 `deliverable_ref=blob:{id}`。
- **GET `/exchanges/{id}/deliverable`**：相关方可取回交付物原文（VDC 仍只含 result_hash）。
- **独立仲裁**：`TROODON_ARBITRATOR=agent_id` 后，`resolve` 仅允许该身份（需鉴权）；
  未设时仍允许相关方裁决（开发模式）。
- **AP2 HTTP**：`HttpAP2Gateway` + `create_ap2_fake_app()`，与 x402 同形 REST。
- **reverify**：`--verify-result` 支持用 `replay_inputs_ref.schema_inline` 独立复跑验收。

### 14.15 内容寻址 blob + 法币网关 + plugfest（2026-06-28）

- **内容寻址**：`deliverable_ref` = `result_hash` = `sha256(canonical(deliverable))`；同内容 dedup。
- **访问控制**：`GET …/deliverable` 仅相关方；`POST …/deliverable/verify` **公开**（只返回 hash 是否匹配，不泄露存储正文）。
- **FiatSettlement**：`HttpFiatGateway` + `create_fiat_fake_app()`；`TROODON_SETTLEMENT=fiat` + `TROODON_FIAT_URL`。
- **Plugfest**：`demo/plugfest.py` — 单节点 + **多节点联邦**（provider 节点交割、client 节点跨节点拉 VDC 独立复验）。

### 14.16 独立 vdcs 表 + witness/质押 v2 占位（2026-06-28）

- **vdcs 表**：`SQLiteStore` 将 VDC 正文存入 `vdcs(vdc_id, exchange_id, data)`；`exchanges.data` 只留 `vdc_id` 引用。
- **引擎**：`find_vdc()` 优先查表；重启后 `get(exchange)` 自动 hydrate。
- **v2 占位**：`troodon/v2/witness.py` + `spec/schemas/witness-attestation.schema.json` / `stake-lock.schema.json`；
  `POST /v2/witness/validate`、`POST /v2/stake/validate` 可校验；attach/lock/slash 默认 **501**（`WITNESS_V2_ENABLED=False`）。

### 14.17 联邦信誉 + 密钥轮换 + 导出 + 配置（2026-06-28）

- **跨节点信誉**：`troodon/v2/federation.py` — `GET /reputation/export`、`POST /v2/reputation/validate`；
  `POST /v2/reputation/import` 默认 **501**（`TROODON_FEDERATION_V2=1` 启用）。
- **密钥轮换占位**：`troodon/key_history.py` + `spec/schemas/key-history.schema.json`；
  `POST /v2/identity/key-history/validate`。
- **Exchange 导出**：`GET /exchanges/{id}/export` — 公开元数据 + VDC；相关方含 `deliverable`。
- **部署默认值**：`NodeConfig.default_timeouts`（`TROODON_TIMEOUT_*`）；`TROODON_WITNESS_V2` 启用 witness attach。
