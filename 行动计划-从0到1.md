> **当下落点**：用陌生软件 Agent 模拟未来物理世界「跨主体、无预建关系」的可验证交换。

## 一、关于「注册公司」：可以，但要摆对位置

| 问题 | 结论 |
|------|------|
| 现在注册公司可以吗？ | 可以，但**保持轻量**（够签合同/开户/持商标/对外可信即可，别背重成本）。 |
| 公司是什么角色？ | **身体**——落地、服务、未来的清算与信誉网络。**不是灵魂。** |
| **铁律** | **公司绝不能拥有协议/规范/商标的最终所有权。** 这些走「个人或公司暂持 → 未来移交中立基金会」的路径，并写进 `GOVERNANCE.md`。 |
| 现阶段公司的真实用途 | 持商标与域名、作为 **CLA 的受让主体**、未来接 grant/投资、对外可信度。 |

> 一句话：**注册公司是为「以后能签约、能接钱、能持资产」做准备；但协议永远归共域。两者分开。**

## 三、里程碑

### 阶段 0 · 地基（先把不可逆的、易被抢的占住）

**A 组织/法律**
- [ ] 定一个**有显著性**的名字 → 查重（商标/域名/GitHub/包名/社媒）
- [ ] **免费占住**：域名、GitHub 组织、社媒、包名
- [ ] 注册**轻量公司**
- [ ] **公开之前**在国内申**文字商标**（中国先申先得、防抢注）
- [ ] 建仓库骨架：
  - `LICENSE`：规范文本 **CC-BY**、参考代码 **Apache-2.0**
  - `GOVERNANCE.md`：写明「当前由本实体主导，里程碑 X 移交基金会」
  - `CONTRIBUTING.md` + **CLA/DCO**（为将来捐基金会留路）
  - `README`：项目是什么、不做什么（不铸币/不绑链/开放）

### 阶段 1 · 规范 v0（B 轨，可一个人写）

- [ ] **资源本体 v0**：先定软件类型 `compute.inference` / `data.*` / `service.task`；**预留**物理 `energy.*` / `actuation.*`（脊椎从软件通向物理）
- [ ] **VDC v0 最小字段**：`who / what / how_much(+结果哈希) / under_rule / evidence(+证据等级) / signatures`
- [ ] **交换状态机**：缔约 → 托管 → 交付 → 验收 → 双签 VDC → 释放/超时退款 → 写信誉
- [ ] **结算适配接口**：可插拔、可 mock、**可对接法币、不绑币**
- [ ] **验收机制 v0**：先用**确定性校验**（schema/哈希），最干净无争议
- [ ] 规范用语义化版本（VDC 0.1）+ 错误码/幂等/重放防护占位

### 阶段 2 · 参考实现 + 模拟舱 demo（C 轨）

- [ ] **两个陌生 Agent**（A=委托方，B=执行方），事先互不认识
- [ ] **参考交换节点**：身份、托管、验收、VDC 签发、信誉日志
- [ ] **跑通完整仪式**：报价 → 托管 → B 交付带签名+哈希 → 确定性验收 → 双签 VDC → 自动释放 / 超时退款
- [ ] **信誉**：只追加的签名日志，**任何人可复验**（不是「相信我」）
- [ ] 结算腿先 mock（留法币适配口）
- [ ] 第一个 demo 任务建议：**结构化数据抽取**（验收 = schema/哈希，无歧义）

## 四、技术栈建议（待你确认）

| 项 | 建议 | 备注 |
|----|------|------|
| 语言 | Python 或 TypeScript | 看你熟哪个 |
| 签名 | Ed25519 | 与业界（XAP/ERC-8004）一致，便于将来互操作 |
| VDC 序列化 | v0 用 **JSON** | 后续可换 CBOR/Protobuf |
| 身份 | keypair / DID-lite | 别一上来上全套 DID |
| 仓库 | monorepo：`spec/` + `sdk/` + `node/` + `demo/` | 一个人好维护 |
